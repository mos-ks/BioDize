# MODEL_RESEARCH.md

Model selection for extracting **handwritten** values from scanned **German pharma batch records** into **structured JSON**, with **bounding boxes** for a review UI, and a planned migration from the OpenAI API to an **on-prem** model for GxP data-control.

_Last updated: 2026-06-26._

---

## 1. Executive recommendation

**Use `gpt-5.4` now; target `dots.ocr` (MIT) — backed by `PaddleOCR-VL-1.5` — as the on-prem swap.** For a 72h hackathon, `gpt-5.4` is the fastest path to a working demo: it natively supports strict `json_schema` Structured Outputs with image input, handles German out of the box, supports `detail="original"` (the single most important flag for handwriting and low-contrast scans), and posted the lowest hallucination rate (~6%) on a handwritten-forms benchmark while costing ~half of `gpt-5.5` ([OpenAI cookbook](https://developers.openai.com/cookbook/examples/multimodal/document_and_multimodal_understanding_tips), [OpenAI models](https://developers.openai.com/api/docs/models)). For GxP / data-sovereignty, the on-prem target is a small, permissively licensed specialist OCR-VLM served via **vLLM's OpenAI-compatible endpoint** so the swap is a `base_url` + model-name change, not a rewrite: **`dots.ocr`** (~1.7B, **MIT**) or **`PaddleOCR-VL-1.5`** (0.9B, **Apache-2.0**, ~2 GB VRAM), both of which emit per-element JSON with bounding boxes natively ([dots.ocr](https://github.com/rednote-hilab/dots.ocr), [PaddleOCR-VL](https://huggingface.co/PaddlePaddle/PaddleOCR-VL)). **Critical caveat:** no open self-hosted model is strong on messy cursive handwriting, and **no general VLM (including GPT) reliably produces precise bounding boxes** — for the review UI, get boxes from an OCR engine's native word/line polygons and use the VLM only to *read* the value (see §3, §5).

## 3. Getting accurate bounding boxes for handwritten values

**The single most important finding: do not ask a general VLM for the box.** OCRBench v2 measures this directly — on English text-spotting (localize + recognize), GPT-4o scored **0** and Qwen2.5-VL-7B **1.2**; text-referring IoU localization stays well under 50 for every VLM tested. One cited model scored 78.3% on position-aware VQA but only **12.9% IoU** on actual region localization — it knows roughly where the value is but cannot draw the box ([OCRBench v2, arXiv 2501.00321](https://arxiv.org/html/2501.00321v2)). GPT-4o/GPT-5-family vision lacks inherent visual grounding ([OpenAI community](https://community.openai.com/t/getting-gpt-vision-to-return-coordinates/671669)); OpenAI's own pattern is a coordinate *contract* in a 0–999 space, explicitly **not pixel-exact**, optionally verified by crop-and-rerun ([cookbook](https://developers.openai.com/cookbook/examples/multimodal/document_and_multimodal_understanding_tips)). Qwen2.5-VL is the best of the bunch but returns **absolute pixels on the *resized* image** (longest side ~1280, snapped to multiples of 28), so you must rescale using `resized_width`/`resized_height`; frameworks that drop those params (e.g. Ollama) produce Y-axis offset errors ([HF discussion](https://huggingface.co/Qwen/Qwen2.5-VL-7B-Instruct/discussions/13)).

**Recommended architecture (highest reliability first):**

1. **Get word/line polygons from an OCR engine that emits boxes natively.** For a cloud-allowed dev phase: **Azure Document Intelligence Read** (`prebuilt-read`, v4.0 GA) returns `{content, polygon, confidence, span}` per word, flags handwriting via a separate `styles` collection (`"isHandwritten": true` + confidence), and prebuilt/custom field extraction returns each value's own `boundingRegions[].polygon` + confidence — the cleanest "value → box" mapping ([Azure Read](https://learn.microsoft.com/en-us/azure/ai-services/document-intelligence/prebuilt/read?view=doc-intel-4.0.0), [analyze-response](https://learn.microsoft.com/en-us/azure/ai-services/document-intelligence/concept/analyze-document-response?view=doc-intel-4.0.0)). **Google Document AI** ("best-in-class handwriting, 50 languages") returns `boundingPoly` with both pixel `vertices` and 0–1 `normalizedVertices` ([handle-response](https://docs.cloud.google.com/document-ai/docs/handle-response)). **AWS Textract** returns per-word/line boxes for printed and handwritten text. For on-prem: **dots.ocr** and **PaddleOCR-VL** emit element-level boxes natively; **Surya** emits char/word/line boxes.
2. **Bind the value to tokens, then union their boxes.** If you use a VLM to *read/interpret* a messy handwritten value (its real strength), **do not trust its coordinates** — string-match the VLM's returned text against the OCR engine's `words[].content` and use the **OCR polygon(s)** as the on-screen rectangle. This pairs the VLM's stronger handwriting *reading* with the engine's accurate *geometry*.
3. **Surface confidence in the UI.** Azure, Google, and Mistral OCR 4 all return per-word/per-field confidence; Mistral OCR 4 is explicitly positioned for human-in-the-loop verification ([Mistral OCR 4](https://mistral.ai/news/ocr-4/)). Route low-confidence handwritten tokens to human review with the box pre-drawn from the OCR polygon.

**For the on-prem end state:** **Mistral OCR 4** is the only *managed-style* engine that bundles boxes + typed blocks + word confidence and offers a licensed single-container self-host — but it is **proprietary (not open weights)** and makes **no explicit handwriting claim**, so it must be validated on real batch records before relying on it ([model card](https://docs.mistral.ai/models/model-cards/ocr-4-0)). The open-weight alternative is `dots.ocr`/`PaddleOCR-VL` for geometry, validated for handwriting on your own scans.

## 4. Local-swap path (OpenAI-compatible `base_url` trick)

Every recommended open model serves an **OpenAI-compatible `/v1` endpoint via vLLM** (most also via SGLang; Qwen via Ollama). This means the migration off the OpenAI API is, mechanically, a **two-line change**: point the SDK at your local server and change the model name. No request/response rewrite, no data leaving the network — which is the whole point for GxP data-control.

```python
# NOW (cloud)
client = OpenAI(api_key=OPENAI_API_KEY)
resp = client.responses.create(model="gpt-5.4", input=[...])

# LATER (on-prem, air-gapped) — start: vllm serve rednote-hilab/dots.ocr --trust-remote-code
client = OpenAI(base_url="http://ocr-host:8000/v1", api_key="not-needed")
resp = client.chat.completions.create(model="rednote-hilab/dots.ocr", input=[...])
```

**Which on-prem model to target** (for regulated, commercial, air-gapped pharma):

- **Primary OCR/extraction:** **`dots.ocr`** (MIT, ~1.7B) or **`PaddleOCR-VL-1.5`** (Apache-2.0, 0.9B, ~2 GB VRAM) — both tiny, permissively licensed, emit structured JSON + bounding boxes, integrated in vLLM (dots.ocr ≥0.11.0 with prebuilt `rednotehilab/dots.ocr:vllm-openai` Docker image; PaddleOCR-VL official since 2025-11-04). **`DeepSeek-OCR`** (MIT, 3B) if you need high-throughput PDF batch on A100-class hardware (~2,500 tok/s on a single A100-40GB).
- **General-VLM fallback / reasoning over documents:** **Qwen3-VL** (Apache-2.0 at every size) at 8B or 32B.
- **Licensing traps to avoid without legal review** — the license **varies by model size** in some families: **Qwen2.5-VL-3B** is `qwen-research` (**non-commercial**); **Qwen2.5-VL-72B** and **InternVL3-78B** inherit the restrictive **Qwen License**. Prefer **Qwen3-VL** and **InternVL3.5** (both Apache-2.0 across all sizes) to sidestep this. Before production, pull the actual `LICENSE` file from the HF repo **at the exact commit you deploy** and have legal confirm — license fields have changed per-revision in the Qwen family ([Qwen2.5-VL-3B](https://huggingface.co/Qwen/Qwen2.5-VL-3B-Instruct), [InternVL3-78B](https://huggingface.co/OpenGVLab/InternVL3-78B)).
- **Mistral self-host nuance:** if you specifically want **Mistral OCR 4 on-prem**, it is a **contractual enterprise single-container license, not an open-weights download** — only pursue it with a signed agreement. The open Mistral vision models (Pixtral / Mistral Small 3.x / Small 4) are Apache-2.0 but are **general VLMs, not tuned OCR/document parsers**, and are weaker on structured extraction than the specialists above.

## 5. Caveats and uncertainties

- **Handwriting is the core risk, and it is under-measured.** No public benchmark isolates **messy handwritten numeric/date-field** accuracy — exactly the fields that matter most in a batch record (a single wrong digit is critical). Frontier VLMs give the best *raw* recognition on messy handwriting (~6–12% CER), **but their advantage comes from context**, and isolated numeric/date fields have none — VLM accuracy drops sharply on low-context/regional crops and they hallucinate without a calibrated confidence score ([arXiv 2503.06472](https://arxiv.org/pdf/2503.06472), [F22 Labs](https://www.f22labs.com/blogs/ocr-vs-vlm-vision-language-models-key-comparison/)). **The production-grade answer is a hybrid: OCR/HTR for detection + bounding boxes + confidence, a VLM to adjudicate low-confidence digits, and rule-based validation** (date formats, range/checksum checks, European "1" vs "7" crossbar). No single tool wins messy numeric fields outright.
- **German is a weak spot for VLMs.** VLMs have a strong English pretraining bias; the peer-reviewed Ströbel et al. benchmark establishes the VLM lead only on **modern** handwriting and finds **no consistent winner on historical** documents ([arXiv 2503.15195](https://arxiv.org/abs/2503.15195)). For modern German *print/neat* handwriting this is usually fine; for **Kurrent/Sütterlin/Fraktur** (possible in old archival batch records) VLMs fail and **Transkribus** dominates (German M1 ~4.7% CER; [model](https://www.transkribus.org/models/transkribus-german-handwriting-m1)).
- **Vendor and aggregator numbers, not independent benchmarks.** Mistral OCR 4's benchmarks (OlmOCRBench 85.20, OmniDocBench 93.07, 72% win rate), its 88.9% handwriting figure (which is actually **OCR 3-era, internal**), and the 170-language/German claims are **Mistral-reported** ([news](https://mistral.ai/news/ocr-4/)). The sub-1.3% IAM CER figures for GPT-5 / Claude Opus 4.7 / Gemini 3 come from a **single aggregator (CodeSOTA)**, not peer review; the best *peer-reviewed* IAM CER (1.23%) is still held by the specialized **HTR-JAND**, so "VLMs clearly win IAM" is a near-tie, not settled. OpenAI pricing/feature figures came from pages that returned partial content and third-party trackers — **confirm on OpenAI's live pricing page before committing**.
- **Bounding-box reliability is the architectural constraint, not a tuning detail.** The OCRBench v2 result (GPT-4o = 0 on text spotting) means a VLM-only bbox will be wrong often enough to undermine the review UI even when the *value* is read correctly. The hybrid "OCR-box + VLM-read" design in §3 is non-negotiable for a usable review experience.
- **EU data residency for German PII.** OpenAI applies a ~10% uplift for regional/data-residency endpoints on models released on/after 2026-03-05 — relevant for German patient/operator PII during the cloud phase, and a further reason the on-prem swap matters for GxP.
- **Unverified / fast-moving items.** Several scout hits referenced speculative future models; the **DeepSeek-OCR 2** license was not explicitly restated (treated as open), and **Mistral OCR 4 handwriting support is unverified** by the vendor. Re-verify any model against its primary HF/GitHub card at deploy time.

### Recommended validation step before committing (within the 72h)
Build a 20–50 page golden set of real (or representative) German batch-record scans with hand-labeled values + boxes, then run **`gpt-5.4` (with `detail="original"` + strict `json_schema`)** against **Azure Read or `dots.ocr`** for the boxes, and measure field-level exact-match on numeric/date fields specifically. This is the only way to de-risk the two weakest-evidenced claims (handwriting accuracy, German performance) before the on-prem swap.
